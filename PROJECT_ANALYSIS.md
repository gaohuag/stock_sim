# 模拟炒股交易引擎 — 项目文档与可行性分析

## 一、项目概述

本项目是一个**全自动模拟量化交易系统**（v2），通过接入天天基金 API 和新浪财经 API 获取实时行情数据，基于预设的价格触发规则自动执行买卖决策，并生成 HTML 报告和邮件日报。

- **初始资金**：约 106,992 元
- **当前总市值**：约 110,933 元（累计收益率约 +3.7%）
- **运行方式**：通过 `run_sim.sh` 定时执行（cron job）
- **通知方式**：邮件发送至 gaohuag@126.com

## 二、交易策略详解

### 2.1 策略核心：分批网格 + 回调买入 + 趋势止盈 + 硬止损

这是一套**多条件触发式规则策略**，不依赖复杂的技术指标，而是靠预设价格阈值触发买卖。核心逻辑如下：

#### 买入逻辑（`check_buy_signals`）

| 条件 | 规则 |
|------|------|
| **回调买入** | 当前价格 ≤ 预设触发价时，按目标仓位比例买入 |
| **追涨买入** | 无持仓 + 价格在触发价 ~ 触发价×1.03 之间，允许追涨（避险资产禁用） |
| **仓位控制** | 每个标的按 `position_size_pct` 占总资产比例计算目标金额，不足 80% 时才补仓 |
| **最小交易额** | 单笔买入不足 1000 元时跳过 |
| **整手约束** | 股票按 100 股整手买入，预留 0.03% 手续费空间 |

#### 卖出逻辑（`check_sell_signals`）

| 条件 | 规则 |
|------|------|
| **止损** | 价格 ≤ `stop_loss` 时全部卖出 |
| **第一止盈位** | 价格 ≥ 第一个目标价 → 卖出持仓的 50%（至少 100 股） |
| **第二止盈位** | 价格 ≥ 第二个目标价 → 全部清仓 |
| **手续费** | 卖出费率 0.13%（含印花税） |

## 三、系统架构

```
stock_sim/
├── simulator.py        # 主引擎：行情获取、信号检测、交易执行、HTML 报告生成
├── email_notify.py     # 邮件发送模块：纯文本 + HTML 双版本日报
├── portfolio.json      # 持仓状态持久化：现金、持仓列表、交易日志
├── watchlist.json      # 股票池配置：触发价、止盈止损、仓位比例
├── report.html         # HTML 持仓报告（卡片式 UI）
├── trades.log          # 逐次执行日志
├── run_sim.sh          # Cron 定时执行脚本
```

### 数据流

```
Cron 定时触发 → simulator.py main()
  ├─ 加载 portfolio.json + watchlist.json
  ├─ 批量获取行情（天天基金 API / 新浪 API）
  ├─ check_buy_signals()  → 生成买入订单
  ├─ check_sell_signals() → 生成卖出订单
  ├─ execute_trades()     → 更新持仓和现金
  ├─ save_json()          → 持久化 portfolio.json
  ├─ generate_html_report() → 输出 report.html
  └─ email_notify.send_report() → 邮件推送日报
```

## 四、可行性分析

### 4.1 策略层面的优势

1. **规则明确、无主观判断**：所有交易由预设价格阈值触发，避免了追涨杀跌的情绪化交易，执行力 100%。

2. **分批建仓 / 分批止盈**：在关键支撑位分 3 批买入，在目标位分 2 批卖出。这是经典的**网格交易思想**，在震荡市中能有效降低持仓成本、锁定利润。

3. **风险控制到位**：每只标的都有明确止损线（约 -10% ~ -15%），单个标的通过仓位比例控制（5%~25%），防止单票重仓失控。

4. **多资产分散**：7 只标的覆盖了 A 股 ETF（半导体、电网、通信、创业板）、港股 ETF（恒生科技）、A 股个股（德业股份）、商品 ETF（黄金），有效分散单一市场和行业风险。

5. **主攻方向聚焦**：65% 仓位集中在科技成长主线（半导体 + 电网 + 通信 + 恒生科技），符合当前 AI 算力基建化、电网智能化的产业趋势，具有较强的投资逻辑支撑。

6. **全自动闭环**：从行情获取 → 信号检测 → 交易执行 → 报告生成 → 邮件推送，形成完整闭环，人工只需查看日报即可。

### 4.2 策略层面的风险与局限

1. **依赖精确价格触发，可能踏空或接飞刀**
   - 策略本质上假设价格会"回调"到预设线附近。在强势单边上涨行情中，价格可能一路上涨从不回调，导致**系统性踏空**。
   - 反之，在单边下跌中，层层触发买入可能导致仓位越加越重、亏损不断扩大（虽然止损可以截断尾部风险，但连续止损会持续消耗本金）。

2. **追涨机制的设置问题**
   - 追涨条件：价格在触发价 100%~103% 之间才追涨，这 3% 的窗口非常窄。
   - 以 159558 为例，当前价 3.6763，第一触发价 3.3，价格已远超 3.3×1.03=3.399，**追涨窗口早已关闭**。这意味着如果一直不回调，将永远无法建仓。

3. **触发价设置偏低，与市场脱节**
   - 半导体 ETF 当前 3.67，触发价设 3.3 以下 → 需下跌约 10% 才触发
   - 德业股份当前 101.44，触发价设 95 → 需下跌约 6.3% 才触发
   - 这些价位距离现价较远，在当前偏多市场环境下，可能长期无法成交。

4. **缺少动态调参机制**
   - 所有触发价、止损价、仓位比例都是静态配置，不会随市场变化或账户盈亏自动调整。
   - 没有类似"移动止损"（trailing stop）来保护已有利润，半导体 ETF 已浮盈 25.7%，但止损仍在 2.85（意味着允许全部利润回吐）。

5. **只做多不做空**
   - 策略纯多头，无法在下跌市场中获利。熊市中只能空仓等待（现金闲置）。

### 4.3 代码层面的问题

#### 🔴 严重 Bug：买入逻辑被死代码阻断

```python
# simulator.py 第 194-195 行
if True:  # 占位保持缩进
    continue   # <--- 这行会直接跳过后续所有买入订单生成！
```

`check_buy_signals()` 中，所有买入信号可以被检测到，但**永远不会生成实际的交易订单**。所有买入触发逻辑（计算买入金额、生成 BUY trade 等）被无条件 `continue` 跳过，相当于**买入功能完全瘫痪**。

从日志可以验证：尽管系统多次检测到买入信号（如 518880 黄金 ETF），但从未生成实际的买入交易记录。

#### 🟡 其他代码问题

- `simulator.py` 中 `send_email_report()` 函数（第 615 行）与 `email_notify.py` 功能重复，且前者存在 `body` 变量未定义的 bug（第 681 行 `body` 被使用但未赋值）。
- 运行日志显示系统在极短时间内（几秒间隔）多次重复执行，可能是 cron 配置过于频繁，造成不必要的 API 调用和资源浪费。
- 异常处理过于宽松：`get_fund_nav()` 和 `get_stock_price_sina()` 捕获所有异常后仅 `pass`，不记录错误详情，使得难以排查数据源问题。
- 黄金 ETF（518880）的止损价与第二买入触发价相同（都是 8.0），这会导致"买入即触发止损"的矛盾。

### 4.4 合规与风险提示

- 系统使用天天基金和新浪财经的公开 API，**存在合规灰色地带**：高频调用可能触发反爬机制或被封 IP。
- 邮件模块中硬编码了 SMTP 密码（明文存储在源码中），存在安全风险。
- 项目定位为"模拟交易"，但接口使用的是真实的实时行情数据，理论上可以切换为实盘交易接口，届时需要券商 API 授权和更严格的合规审查。

## 五、改进建议

### 短期修复（立即）

| 优先级 | 问题 | 建议 |
|--------|------|------|
| P0 | 买入死代码 | 删除 `simulator.py` 第 194-195 行的 `if True: continue` |
| P0 | `send_email_report` 中 `body` 未定义 | 将 `body` 改为已生成的文本内容，或直接删除重复函数 |
| P1 | 黄金 ETF 止损=买入触发价 | 将止损下移（如 7.8）或买入触发价上移 |

### 中期优化

| 方向 | 建议 |
|------|------|
| **动态触发价** | 根据当前价格和市场状态定期调整触发价，避免长期无法触发 |
| **移动止损** | 对已有利润的仓位（如半导体 ETF +25%）设置 trailing stop，锁定部分利润 |
| **仓位再平衡** | 定期检查各标的实际仓位占比，偏离目标 ±5% 时自动再平衡 |
| **回测系统** | 接入历史数据对策略做回测，验证不同市场环境下的表现 |
| **日志增强** | 完善异常日志记录，增加行情获取失败的重试+告警机制 |
| **Cron 控制** | 将执行间隔调整为合理频率（如每 30 分钟），避免无意义重复执行 |

### 长期方向

| 方向 | 建议 |
|------|------|
| **引入趋势过滤** | 加入均线或市场广度指标，在大盘破位时暂停买入 |
| **做空/对冲** | 引入融券或期权对冲，在下跌市中也能盈利 |
| **ML 辅助** | 用机器学习预测最佳触发价或仓位动态调整 |
| **实盘化** | 对接券商 API（如华泰、东财），从模拟升级为实盘 |

## 六、总结

该项目是一套**设计理念合理但实现存在关键 bug 的规则式量化交易系统**。策略本身——分批建仓 + 分批止盈 + 严格止损——是一种经过验证的稳健交易方法论，尤其在震荡市场中表现较好。

**当前最大的风险不是策略逻辑，而是代码 bug**：买入逻辑被死代码阻断，导致系统实质上是一个"只卖不买"的半残状态。修复这个问题后，系统理论上可以正常运行。

策略的主要局限性在于**缺乏动态适应能力**——固定的价格触发点在趋势市中容易踏空，在极端下跌中又可能累积亏损。建议在修复 bug 之后，逐步加入动态调参和回测能力，让系统从"规则执行器"进化为"策略引擎"。
